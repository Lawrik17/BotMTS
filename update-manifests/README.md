# Как обновлять — куда и что загружать

Репозиторий: **github.com/Lawrik17/BotMTS** (если другой — поправь ссылки в manifest-файлах и в настройках).

## Главная идея (чтобы URL в настройках вписать ОДИН раз)

- **`manifest.json`** лежит в этом репозитории (папка `update-manifests/`). Его raw-ссылка **постоянная** —
  её и вписываешь в настройки один раз. Каждый релиз ты просто редактируешь этот файл (версию + ссылку на zip).
- **zip пакета** и **`Updater.exe`** — тяжёлые, поэтому кладутся в **GitHub Release** (не в репозиторий).

Одна раскладка `Updater.exe` подходит для всех трёх приложений — он не зависит от приложения.

---

## Разовая настройка (один раз вписать ссылки)

Открой настройки и вставь постоянные raw-ссылки на манифесты:

| Что обновляем | Где вписать ссылку | Ссылка (raw) |
|---|---|---|
| Клиент 6D6X6 | BotStats → «URL манифеста обновлений 6D6X6» | `https://raw.githubusercontent.com/Lawrik17/BotMTS/refs/heads/main/update-manifests/manifest.6D6X6.json` |
| Сервер BotStats | BotStats → «URL манифеста обновлений BotStats» | `https://raw.githubusercontent.com/Lawrik17/BotMTS/refs/heads/main/update-manifests/manifest.BotStatsMTS.json` |
| AdminClient | AdminClient → «Пакетное обновление» → «URL манифеста AdminClient» | `https://raw.githubusercontent.com/Lawrik17/BotMTS/refs/heads/main/update-manifests/manifest.AdminClient.json` |

## Каждый релиз (когда вышла новая версия приложения)

1. **Подними версию** в проекте (`AssemblyVersion`/`FileVersion`) — иначе апдейтер решит, что обновлять нечего.
2. **Собери Release** и возьми папку publish:
   - BotStats / AdminClient (net8, AnyCPU): их `bin\Release\net8.0-windows\` либо отдельный publish-каталог;
   - 6D6X6 (net8, x64): `Y:\6D6X6\6D6X6\bin\x64\Release\net8.0-windows\` либо отдельный publish-каталог.
     В нём уже должны находиться `launcher`, `Apps` и `Service\6D6X6.Service.exe`.
3. **Заархивируй СОДЕРЖИМОЕ папки в zip** — так, чтобы `AppName.exe` и все `.dll` были в **корне архива**
   (не внутри вложенной папки). Пример имени: `BotStatsMTS-1.0.2.zip`.
4. На GitHub: **Releases → Create a new release**, придумай тег (например `botstats-1.0.2`), и **приложи файлы**:
   - zip пакета,
   - автономный `Updater.exe` (лежит `Y:\6D6X6\Updater\bin\x64\Release\net48\Updater.exe`; кладём один раз, дальше можно
     переиспользовать ссылку со старого релиза).
5. Скопируй прямые ссылки на приложенные файлы (правой кнопкой по файлу в релизе → «Copy link»).
6. **Отредактируй manifest** нужного приложения в папке `update-manifests/` и запушь в `main`:
   - `version` — новая версия (та же, что подняли в п.1),
   - `downloadUrl` — ссылка на zip,
   - `updaterUrl` — ссылка на `Updater.exe`.
7. Готово. Ссылка в настройках менять не надо — она указывает на manifest, а его ты только что обновил.

Готовые шаблоны для всех приложений находятся в `update-manifests/examples/`:

- `manifest.6D6X6.example.json`;
- `manifest.BotStatsMTS.example.json`;
- `manifest.AdminClient.example.json`.

SHA-256 готового zip вычисляется командой:

```powershell
(Get-FileHash -Algorithm SHA256 -LiteralPath 'C:\Release\6D6X6-0.2.5.0.zip').Hash.ToLowerInvariant()
```

Полученную 64-символьную строку нужно вставить в `sha256`. Поле `version` должно в точности совпадать с
`FileVersion` главного EXE внутри zip. `minimumVersion` — самая старая версия, которую разрешено обновить
автоматически; если ограничение не нужно, поле можно оставить пустым. `files` может быть пустым: целостность
всего архива всё равно защищает общий `sha256`.

## Как запустить обновление

- **Клиенты 6D6X6:** BotStats → список ПК → у нужного ПК пункт «Обновить клиент 6D6X6» (или бейдж «обнови»,
  когда версия ниже целевой). То же самое доступно в AdminClient (список общий).
- **Сервер BotStats:** BotStats → настройки → «Обновить BotStats». Или из AdminClient → настройки →
  «Обновить сервер BotStats».
- **AdminClient:** AdminClient → настройки → «Обновить AdminClient».

Приложение само скачает пакет, сделает бэкап, поставит и перезапустится.

Для 6D6X6 папка `Service` обновляется тем же архивом. Перед заменой файлов updater запоминает состояние
`6D6X6Service`, останавливает службу, заменяет пакет и запускает её снова только в том случае, если она работала
до обновления. При rollback предпринимается попытка вернуть прежние файлы и прежнее состояние службы.
Операция требует прав администратора; без них обновление прерывается до замены файлов.

## Что НЕ затирается при обновлении

Настройки и данные в списке исключений апдейтера: `SettingsBot.ini`, `settings.json`, `admin_settings.json`,
`cookies.dat`, `AppSettings.ini`, локальные файлы аккаунтов AutoLogin, БД игр/сессий, логи и папки
`Games`, `backup`, `logs`.

Папки `launcher`, `Apps` и `Service` **не исключены**: их код и зависимости обновляются целиком из архива.
Внутри `Apps`/`launcher` отдельные защищённые имена (настройки, аккаунты, `Updater.exe`, `update.7z`) сохраняются.

Установка сейчас добавочная: файл, которого больше нет в новом zip, автоматически не удаляется со старой
установки. Если релиз должен удалить устаревший модуль, его надо внести в отдельный контролируемый список
удаления будущего манифеста, а не рассчитывать на отсутствие файла в архиве.

## Перед первым реальным релизом

Текущие `downloadUrl`/`updaterUrl` в шаблонных manifest-файлах указывают на ещё не опубликованные assets и
до создания соответствующего GitHub Release возвращают 404. Также в рабочий manifest нужно записать SHA-256
zip-пакета и, желательно, список SHA-256 файлов. Без опубликованного release обновление из интерфейса запустить
нельзя.
